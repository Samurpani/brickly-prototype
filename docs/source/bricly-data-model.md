# Bricly Data Model — Ideal Product (v1)

32 core entities. Entities 1-27 are the core ideal-product model. Entities 28-32 are the **post-sale digitisation layer**, marked as future expansion. They are sketched at the same level of detail as the rest so the data model can accommodate them without refactoring when the time comes, but they are explicitly out of MVP and v1 scope. The data model is the product. This is the **ideal-product** model, not the MVP. MVP scoping (which entities, states, and relationships ship in v1 of the product itself) is a separate exercise that strips back from this foundation.

Every entity below has a defined shape, defined relationships, and defined states. This is the foundation the capability surface, API, MCP server, and UI all sit on top of.

---

## 1. Workspace

The top-level tenant. The developer company. Every other entity belongs to a Workspace.

**Key attributes**
- Name, legal entity, primary market
- Subscription tier and billing
- CorporateBrand cluster (logo, primary palette, typography, brand guidelines URL, brand inheritance default for new projects)
- Default settings (currency, language, locale, hold expiry rules, commission templates)
- Onboarding completion status

**Relationships**
- HAS_MANY: Users, Projects, Partners, Contacts, Campaigns
- One per tenant. Everything is scoped under this.

**States**
- Trial, Active, Past_Due, Suspended, Cancelled

---

## 2. Project

A development the workspace is bringing to market. Container for everything specific to that build: units, constraint model, brand kit, assets, campaigns, opportunities.

**Key attributes**
- Name (set at onboarding or generated by brand consultation Mode A part 1 based on location and project-type research), location (address, plot reference, coordinates), market
- Status, target launch date, target sell-out date, completion date
- Total unit count, target revenue, current sell-out percentage
- Project type (single block, multi-block, mixed-use, etc.)
- Source files reference (architect drawings, CAD, site studies, surrounding context studies, plot photos)
- Style tags
- BrandKit reference (the chosen one), ConstraintModel reference (current version), ContentBackbone reference (current version, post-launch-package generation)
- Personas referenced (the buyer audiences the project is being marketed to)
- Public-facing project description, microsite URL, availability list URL

**Relationships**
- BELONGS_TO: Workspace
- HAS_ONE: ConstraintModel (current version, with version history), BrandKit (chosen, after brand selection step), ContentBackbone (after launch package generation Phase 3)
- HAS_MANY: Units, Buildings (for multi-block projects, each Building has its own name), Assets, Campaigns, Opportunities (via Units), Personas, FinishPackages
- HAS_MANY: Briefs (the launch package and any subsequent generation requests target the Project)

**States**
- Draft, Briefed, Branding, Generating, In_Review, Launching, Selling, Sold_Out, Completed, Archived

Project state transition from Launching to Selling is not part of the launch package generation flow. It is owned by the separate Go-to-Market flow (developer's explicit decision to go live).

---

## 3. Unit

A single sellable property within a Project. The atomic inventory item.

**Key attributes**
- Unit ID, type (apartment, penthouse, townhouse, etc.), floor, beds, baths
- Internal sqm, external sqm, total sqm, orientation, view
- Canonical spatial position: building reference (for multi-block projects), side or facade reference, level, position on level (e.g., corner, mid, end), orientation
- Unit label as used on architect drawings (the string used in CAD, for matching during extraction)
- Price, payment terms, completion date
- Layer 2 ConstraintModel references (available finish options, layout variants, optional configurations for this unit)
- Floor plan asset reference (the published one)
- Current Opportunity reference (when held, reserved, or sold)
- Sold-to Contact reference (when sold)

**Relationships**
- BELONGS_TO: Project
- HAS_MANY: Opportunities (active and historical, including "considered but not chosen")
- HAS_ONE: bound Opportunity (when not Available)

**States**
- Available, Held, Reserved, Sold, Off_Market, Snagging, Handed_Over

---

## 4. Contact

A person known to the workspace. Universal person record used for prospects, active clients, past clients, referrers, agency contacts, and external professionals (notaries, lawyers, etc.). Segmentation by purpose is handled by Contact attributes, state, and UI views (tabs, smart lists), not by separate entities.

**Key attributes**
- Identity: first name, last name, email, phone, WhatsApp, nationality, languages
- Contact type / role tags: prospect, active_client, past_client, referrer, agency_contact, external_professional (notary, lawyer, finance broker, surveyor), other. Multi-valued, can be more than one over a lifetime.
- Default preference profile: client type (investor, owner-occupier, expat, etc.), budget min/max, preferred location, finance method, timeline, lifestyle notes
- Capture cluster (filled in on first capture, persists forever): source Campaign, source Partner, capture timestamp, capture context (which page, which form, which agent session), capture channel (form, MCP, walk-in, agent introduction, manual entry), original message or enquiry content, qualification score (computed by agent), dedupe candidates identified at capture, conversion event timestamp (when moved from Lead state to Qualified or beyond)
- KYC status, ID document references, proof of funds references
- Communication preferences (preferred channel, opt-in status, frequency)
- Owner User

**Relationships**
- BELONGS_TO: Workspace
- HAS_MANY: Opportunities (across multiple Pipelines, concurrent and historical), Activities, Appointments, Documents
- OPTIONAL: linked Partner (if introduced by an agency or referrer)

**States**
- Lead, Qualified, Active, Past_Client, Disqualified, Spam, Duplicate, Cold, Do_Not_Contact

---

## 5. Opportunity

One Contact's presence in one specific Pipeline. A Contact can have multiple concurrent Opportunities across different pipelines (e.g., simultaneously in a marketing nurture pipeline and a sales pursuit pipeline). Each Opportunity has its own owner, stage, attached entities, and lifecycle. For sales pipelines, an Opportunity can have multiple Units attached in early stages, collapsing to one bound Unit at Hold.

**Key attributes**
- Contact reference
- Pipeline reference (which Pipeline this Opportunity sits in)
- Current stage (must be valid for the Pipeline's state machine)
- Assigned User (owner, role depends on pipeline type: sales rep for sales pipeline, marketing lead for marketing pipeline, etc.)
- Per-opportunity preference profile (overrides Contact defaults for this pursuit)
- Attached entities (polymorphic, depends on Pipeline type):
  - Sales pipeline: shortlisted Units, bound Unit (when Hold placed onwards), historical "considered but not chosen" Units
  - Marketing pipeline: typically no Unit attachment, optionally Project of interest
  - Pre-buy land pipeline: Plot or Site references (future)
  - Other pipeline types: as defined per Pipeline
- Source: lead source type, originating Campaign, originating Partner, source notes
- Commission terms (sales pipeline only): agent split, referrer/partner split, agency commission
- Stage history with timestamps
- Reservation terms, deposit amount, PaymentPlan reference (sales pipeline only)
- Notary appointment reference, signed contract Document reference (sales pipeline only)

**Relationships**
- BELONGS_TO: Contact, Workspace, Pipeline
- ASSIGNED_TO: User (owner role depends on pipeline)
- HAS_MANY: Activities, Appointments, Assets (personalised collateral), Briefs (personalisation briefs), ApprovalRequests
- HAS_MANY (sales pipeline only): Units (shortlisted + considered + bound), PaymentPlans
- OPTIONAL: Partner (referrer or agency), Campaign (attribution)

**States**
- Stages are defined by the linked Pipeline, not hardcoded here. The Pipeline's state machine determines valid stages and transitions. The Opportunity carries the current stage as a reference to a stage within the Pipeline definition.
- Universal lifecycle states (orthogonal to pipeline stage): Active, Paused, Closed_Won, Closed_Lost, Archived

## 6. Activity

A logged interaction or event against a Contact, Opportunity, Unit, or Asset. The system's memory of what happened.

**Key attributes**
- Type (call, message, email, note, status_change, system_event, agent_action, tool_invocation)
- Subject reference (polymorphic: Contact, Opportunity, Unit, Asset, Project)
- Actor (User, Agent, External_System)
- Timestamp, channel, content, metadata
- Linked Activities (for threading or causal chains)

**Relationships**
- BELONGS_TO: Workspace
- LINKS_TO: any addressable entity in the system

**States**
- Logged (default, immutable), Pending (for actions awaiting confirmation), Failed (for failed agent actions)

---

## 7. Appointment

A future-dated commitment between one or more parties tied to a deal lifecycle. Viewings, notary signings, contract meetings, handover walkthroughs.

**Key attributes**
- Type (viewing, notary, contract_signing, payment_meeting, handover, other)
- Scheduled time, duration, location (physical or virtual), notes
- Attendees: Users, Contacts, external participants
- Subject: Opportunity reference (primary), Unit reference (optional)
- Reminder rules, calendar sync references (external calendar IDs)
- Outcome notes, follow-up Activity references

**Relationships**
- BELONGS_TO: Workspace
- ATTACHED_TO: Opportunity, Unit (optional)
- HAS_MANY: User attendees, Contact attendees

**States**
- Scheduled, Confirmed, Completed, No_Show, Cancelled, Rescheduled

---

## 8. ApprovalRequest

A pending state change that requires authorisation. Polymorphic across hold requests, commission splits, collateral approvals, exception terms.

**Key attributes**
- Type (hold, reservation, commission_split, collateral, contract_terms, price_exception, other)
- Target reference (polymorphic: Unit, Opportunity, Asset, Commission record)
- Proposed change payload
- Requester User, approver User(s), required approvals count
- Decision, decision reason, decision timestamp
- Expiry time (for stale requests)

**Relationships**
- BELONGS_TO: Workspace
- TARGETS: any approvable entity
- REQUESTED_BY: User
- APPROVED_BY: User(s)

**States**
- Pending, Approved, Rejected, Expired, Withdrawn

---

## 9. Partner

An external party who refers, sources, or co-sells deals. Individual referrers and real estate agencies share this entity with a type discriminator.

**Key attributes**
- Type (individual_referrer, agency)
- Name, contact details, company name (for agencies)
- Default commission terms, agreed attribution rules
- Linked individuals within the partner (for agencies, the agency's reps)
- MCP connection status, API token reference (if connected programmatically)
- Permission scope (what data the partner can access)
- Total commission paid, total deals attributed, performance metrics

**Relationships**
- BELONGS_TO: Workspace
- HAS_MANY: Opportunities (attributed), Contacts (referred or sourced), Activities
- OPTIONAL: linked Contact records for individual referrers

**States**
- Pending_Setup, Active, Paused, Terminated, Disputed

---

## 10. User

A person inside the workspace. Sales rep, sales manager, marketing lead, ops, developer, admin.

**Key attributes**
- Identity: name, email, phone, profile photo, language
- Role (rep, manager, marketing, ops, developer, admin)
- Permission scope (which Projects, which entity types, which actions)
- Assignment rules (which lead sources, which Projects auto-assign to them)
- Performance metrics: targets, current attainment, conversion rates, commission earned, commission pending
- Calendar integration references, messaging channel references

**Relationships**
- BELONGS_TO: Workspace
- ASSIGNED_TO: Opportunities, Contacts, Projects, Appointments, ApprovalRequests (as requester or approver)
- HAS_MANY: Activities (as actor)

**States**
- Invited, Active, Inactive, Suspended, Removed

---

## 11. ConstraintModel

The structured JSON representation of a Project's architectural and editorial constraints. The proprietary middleware that every generated asset respects. Three-layer: hard architectural constraints, developer-defined options, per-Opportunity selections.

**Key attributes**

*Layer 1 — Hard constraints (architect's truth, immutable except by re-extraction):*
- Site: plot boundary, slope, orientation, setbacks, height limits, regulatory constraints, surrounding context references
- Building envelope: footprint, total height, floor count, geometry
- Per-Unit constraints: sqm, aspect, view, floor, structural layout rules
- Floor plan layout rules (e.g., bathrooms only on exposed walls or internal shafts)
- Approved exterior materials and colour ranges
- Plot reality references: real photos of street, plot, surrounding buildings
- Source file references (CAD, drawings, studies)

*Layer 2 — Developer options (editorial menu, versioned, project-wide or per-Unit):*
- Approved interior finish palette (the bounded set buyers can choose from)
- FinishPackage entries (named bundles of materials and finishes, e.g., Package A / B / C) — see entity 14
- Layout variants the developer is willing to deliver
- Optional configurations (study vs second bedroom, kitchen layouts, etc.)

*Layer 3 — Per-Opportunity selections (stored on Opportunity-Unit link, referenced here for completeness):*
- Buyer's chosen variant from the menu, scoped to one Opportunity for one Unit

**Edit permissions**
- Layer 1: locked, only re-extraction or architect-approved updates
- Layer 2: developer or marketing lead
- Layer 3: rep or buyer (within Layer 2 bounds)

**Relationships**
- BELONGS_TO: Project
- VERSIONED: many versions, one current
- REFERENCED_BY: every Brief, every Asset generation, every Unit's available options

**States**
- Extracting, Pending_Review, Approved, Stale (architect issued revised drawings), Deprecated (superseded by newer version)

---

## 12. Asset

A generated output from Studio. Polymorphic across renders, brochures, floor plan graphics, social posts, logos, videos, one-pagers, microsites, availability PDFs. Universal parent with type-specific extensions.

**Universal attributes (Asset parent)**
- Type discriminator
- Source discriminator: Generated (Bricly-produced) or Imported (developer-uploaded)
- File reference (storage URL), format, dimensions, file size
- Version, parent Asset (for variants and regenerations)
- Project reference, Unit reference (optional, for unit-specific assets), Opportunity reference (optional, for personalised assets)
- Source Brief reference (Generated Assets only)
- BrandKit version applied, ConstraintModel version applied
- Generation metadata: tool used (Midjourney, Canva, etc.), generation time, cost
- Vision verification result: attributes checked, per-attribute pass/fail, confidence scores, override status (developer approved despite a flagged discrepancy), audit trail. Informational only on Imported Assets, gating on Generated Assets.
- Approval status, published status
- Channels where currently distributed (website, brochure pack, social, ad campaign, agent share)

**Type-specific extensions**
- Render: camera angle, time of day, weather, view target, interior or exterior, finish selections applied, building face (for multi-face buildings), phase scope (phase_0, phase_2a, phase_2b, phase_2c_variant)
- Brochure: page count, language version, print or digital, included Units, format spec (CMYK + bleed for print, RGB + hyperlinked + QR for digital)
- FloorPlanGraphic: Unit reference, level, branded layer applied, view variant (unit-floor-plan or level-floor-plan-with-unit-highlighted)
- ViewFromLevel: level reference, orientation, source (uploaded vs generated). Surfaced on unit detail page based on Unit.level.
- FaceTransitionVideo: from_face, to_face, duration, source video tool (e.g., Kling), associated exterior render Assets
- MarketingTemplate: platform/format (Meta feed, Meta story, Google display size, Instagram story, reel placeholder), Canva file reference, brand-applied state
- SocialPost: platform, dimensions, caption copy, hashtags
- Logo / Brand asset: format pack contents
- Video: duration, resolution, scenes referenced
- OnePager: target Unit, target buyer profile, language

**Relationships**
- BELONGS_TO: Project (always), Unit (often), Opportunity (when personalised)
- GENERATED_FROM: Brief (Generated source only)
- DISTRIBUTED_VIA: Campaigns, microsite, brochure pack, agent shares

**States**
- Generating, Generated, In_Review, Approved, Published, Stale (underlying data changed), Retired, Failed, Imported (uploaded by developer, bypasses generation)

For Imported Assets: state begins at Imported, transitions to Approved (auto, unless flagged), then optionally Published. Stale flagging still fires on underlying data changes but no automatic regeneration is possible (developer must manually replace).

## 13. BrandKit

The visual identity of a Project. Logo suite, colour palette, typography, brand guidelines. One per Project (the chosen one), generated by Studio from the project brief.

**Key attributes**
- Project reference
- Source: Generated (Bricly-produced through consultation) or Imported (developer-uploaded existing brand)
- Logo suite (primary, secondary, monochrome variants in AI, PNG, SVG, EPS)
- Colour palette (primary, secondary, accents with hex, RGB, CMYK, Pantone, named roles)
- Typography (headline font, body font, optional caption font, weight system, type scale, font files or licence references)
- Brand guidelines PDF reference
- Corporate brand inheritance setting (ignore, loose_reference, must_align)
- Tone of voice (style of the brand itself plus who it's speaking to, grounded in Personas)
- Brand keywords, brand do's and don'ts
- Art direction (structured object: mood keywords, lighting profile per shot type, lifestyle population rules, composition rules per shot type, colour grading profile). Becomes input to every Brief from Phase 2 render generation onwards.
- Application mockups (two: one over project render, one over letterhead / business card / social templates)

**Relationships**
- BELONGS_TO: Project (one current per Project)
- REFERENCES: Workspace.CorporateBrand (when inheritance is set), Persona entities (BrandKit references the Project's Personas, the buyer audience the brand speaks to)
- REFERENCED_BY: every Asset (which BrandKit version was applied)

**States**
- Generating, Candidate, Selected, Refining, Approved, Active, Superseded (replaced by new version), Imported (uploaded by developer, bypasses Generating → Candidate → Selected → Approved, may have gaps that trigger targeted consultation)

For Imported BrandKits: if any of the attributes above are missing (no personas, no tone of voice, no art direction), Bricly runs a targeted ConsultationSession to fill the gaps. Developer always ends with a complete BrandKit.

## 13a. Persona

A structured representation of a buyer audience the project is being marketed to. Generated in the brand consultation Mode A part 1 (or imported if the developer brings existing persona research). Grounds tone of voice and informs campaign targeting, opportunity matching, and copy generation.

**Key attributes**
- Project reference
- Source: Generated (consultation-produced) or Imported (developer-uploaded)
- Name (e.g., "International Investor", "Local Owner-Occupier", "Returning Expat")
- Demographic profile (age range, life stage, household composition, geographic origin)
- Financial profile (income band, investment horizon, financing pattern)
- Motivations (why they're buying off-plan: investment, lifestyle, relocation, downsizing, family)
- Channel preferences (which channels they engage on, which collateral types they value)
- Tone of voice notes specific to this persona (how this audience expects to be spoken to)
- Objections (what typically gives this persona pause about an off-plan purchase)

**Relationships**
- BELONGS_TO: Project
- REFERENCED_BY: BrandKit (informs tone of voice), Brief (copy generation targets persona), Campaign (audience targeting), Opportunity (persona match on lead capture)

**States**
- Generating, Approved, Active, Archived

A Project typically has 2 to 4 Personas. The brand's tone of voice spans all of them; campaign-level and personalised-collateral targeting can scope to one.

---

## 14. Brief

A structured generation request. The input record passed to the orchestration layer. Three flavours: initial project brief, personalisation brief (for a specific Opportunity), individual asset brief or regeneration brief.

**Key attributes**
- Type (project_initial, personalisation, single_asset, regeneration)
- Phase scope (where in the launch package generation flow this Brief sits: phase_0_consultation_pack, phase_2a, phase_2b, phase_2c_variant, phase_2d_transition, brochure_print, brochure_digital, website, marketing_template, launch_grid, or null for non-launch Briefs)
- Target Project reference, target Opportunity reference (for personalisation), target Asset reference (for regeneration)
- Inputs: selections, parameters, copy direction, style tags
- Language (default per Project, additional languages on demand)
- ConstraintModel reference (which version applies)
- BrandKit reference (which to apply)
- Persona reference(s) (which audience the output targets)
- FinishPackage reference (for finish-package-scoped variant generation)
- Requested output types (renders, brochure, one-pager, etc.)
- Requester User or Agent
- Orchestration plan (which external tools will be called: Midjourney, Canva, Kling, etc.)
- Regeneration counter (number of automatic regenerations attempted on vision verification failure, capped at 3 before internal escalation)
- Resulting Assets references

**Relationships**
- BELONGS_TO: Workspace
- TARGETS: Project, Opportunity, or Asset
- REFERENCES: ConstraintModel version, BrandKit version, Persona, FinishPackage
- PRODUCES: one or more Assets

**States**
- Drafted, Submitted, In_Progress, Awaiting_Verification, Completed, Failed, Escalated_To_Internal (verification failed three times, in Bricly internal queue), Approved, Archived

---

## 14a. FinishPackage

A named bundle of interior materials and finishes that the developer offers as a choice to buyers. Lives within the ConstraintModel layer 2 conceptually but treated as a first-class entity so it can be referenced by Briefs, Assets, and Opportunities cleanly. Drives the variant render generation in Phase 2c of the launch package and the FinishPackage selector on the website and in sales meetings.

**Key attributes**
- Project reference
- ConstraintModel version reference (which version this package belongs to)
- Name (e.g., "Light", "Mid", "Dark", "Coastal", "Urban")
- Description (short copy used in marketing collateral)
- Material specifications: flooring, wall finishes, kitchen cabinetry and counters, bathroom finishes, fixtures, joinery, any other developer-defined material slots
- Reference imagery (style images, material samples)
- Default flag (one FinishPackage may be marked as default for the project)
- Price impact (optional, if packages carry different pricing)
- Available on Units (default all, can scope to specific Units or Unit types)

**Relationships**
- BELONGS_TO: Project
- REFERENCED_BY: Unit (which packages available), Asset (variant renders generated per package), Opportunity (buyer's chosen package via Opportunity-Unit link in ConstraintModel layer 3)
- VERSIONED: with ConstraintModel; layer 2 versioning applies

**States**
- Draft, Active, Superseded, Retired

A Project typically has 2 to 4 FinishPackages. The Phase 2c variant render set generates one full interior render set per Active FinishPackage.

## 14b. ContentBackbone

The single content source that seeds both brochure and website at launch. References entities (Project, Building, Unit, FinishPackage, BrandKit, Persona) rather than duplicating them, so structural data flows through the strict cascade. Editorial content lives on the backbone and forks per surface on developer edit.

**Key attributes**
- Project reference
- Project story: name reference, tagline, narrative paragraph (editorial)
- Location content: address, neighbourhood description, points of interest list, accessibility notes (editorial; structural data such as coordinates lives on Project)
- Development content: architectural concept, key features (editorial)
- Amenities: per amenity, name, description, image reference
- Residence types content: per type, short description, button-text and link target conventions
- Specifications content: construction quality, materials narrative, sustainability story
- Investment content: payment terms narrative, completion timeline copy
- Developer / architect copy
- Contact copy
- Lead capture form configuration (which fields, on which surfaces)
- Section toggle state per surface (which sections appear on the website, which on the brochure)
- Fork map: per editorial field, which surfaces have forked from the backbone (Shared, Forked_Brochure, Forked_Website, Surface_Specific)

**Relationships**
- BELONGS_TO: Project (one current ContentBackbone per Project)
- REFERENCES: Project, Building, Unit, FinishPackage, BrandKit, Persona
- CONSUMED_BY: Microsite (project_public_site, embed types), Asset (brochure), and any downstream marketing material that pulls editorial copy
- VERSIONED: editorial edits create new versions; structural changes flow through references, not backbone versions

**States**
- Drafting (during launch package generation), Active, Editing (developer-in-progress edits), Archived

Default fork behaviour: editorial edits on a surface fork from the backbone. Developer can explicitly propagate a fork back to the backbone (and to the other surface) via a "push to website too" action. Structural data (price, status, unit count, FinishPackage definitions) never forks; it always flows through references and the strict cascade.

## 15. Campaign

A coordinated marketing push. Channel-specific creative running over a defined window, attracting Leads, with attached Assets and attributed Opportunities.

**Key attributes**
- Name, Project reference (optional, can be cross-project)
- Channel (Facebook, Instagram, Google, portal, event, email, organic)
- Status, start date, end date
- Target audience description, geographic targeting
- Attached Assets (the creative running as the campaign)
- Budget (optional, MVP can defer), spend to date, cost per lead, ROI
- Attributed Leads / Opportunities count and value

**Relationships**
- BELONGS_TO: Workspace
- REFERENCES: Project (optional), Assets (the creative)
- ATTRIBUTED_FROM: Opportunities (via source field)

**States**
- Draft, Scheduled, Live, Paused, Ended, Archived

## 16. AssetUnitMap

The join entity between an Asset (exterior render, level floor plan, aerial view) and the Units visible on it. Holds the polygon coordinates and interaction metadata that power the interactive sales experience: hover, click-to-explore, deep links into unit detail. Also the structure populated by the floor-plan-to-availability auto-mapping pipeline.

**Key attributes**
- Asset reference (the render, level floor plan, or aerial view)
- Asset context (which Asset type is being mapped: exterior_render, level_floor_plan, aerial, contextual) — informs which UI surfaces consume the map
- Unit reference (the unit being mapped)
- Polygon coordinates (the shape of the unit on this specific Asset's pixel space)
- Hover label, click action, deep link target (e.g., Unit detail microsite, personalised pack URL)
- Conditional display rules (e.g., show only if Unit is Available, hide if Sold)
- Auto-mapped flag (true if extracted by the pipeline, false if manually placed)
- Extraction confidence score (when auto-mapped)
- Reserved for future: `zone_type` (unit | room | feature) to support sub-unit zones later without schema change

**Relationships**
- BELONGS_TO: Workspace
- JOINS: Asset to Unit (many Units per Asset, many Assets per Unit)
- BELONGS_TO: Project (derived via Asset and Unit, both must belong to same Project)

**States**
- Auto_Mapped_Pending_Review (extraction confidence below threshold, needs human confirmation)
- Mapped (active, displayed on rendered Asset)
- Hidden (mapped but suppressed by display rule)
- Stale (underlying Asset regenerated, polygons need re-mapping)
- Removed

The pattern applies to both exterior renders (powering the website availability list's split-view) and level floor plans (powering the unit detail page's "where on the level" view). Multi-face buildings produce one AssetUnitMap per (face_exterior_render, unit) pair; the website switches the active render based on user interaction and the polygon set follows.

## 17. Pipeline

A configurable workflow definition. Defines the stages, transitions, ownership rules, and automation for a class of work. Each Pipeline is owned by a Workspace and can be customised per developer. Opportunities reference a Pipeline and carry a current stage within that Pipeline.

**Key attributes**
- Name, description
- Pipeline type discriminator (sales, marketing, pre_buy_land, post_sale, partner_referral, agency, custom)
- Stage definitions (ordered list): each stage has name, position, default ownership role, expected duration, completion criteria, optional approval requirement
- Valid stage transitions (which stages can move to which)
- Default assigned User role (sales_rep, marketing_lead, ops, etc.)
- Automation rules (e.g., "when an Opportunity enters Closed_Won in the sales pipeline, pause linked marketing-pipeline Opportunities for the same Contact")
- Attached entity rules (which entities can attach to Opportunities in this pipeline: Units, Plots, none)
- Commission triggers (which stage transitions trigger Commission records)

**Relationships**
- BELONGS_TO: Workspace
- HAS_MANY: Opportunities (currently in this pipeline)

**States**
- Draft, Active, Paused, Archived

---

## 18. PaymentPlan

The full payment schedule for a sold or reserved unit. Templated from standard developer plans, instantiated per Opportunity, can be modified mid-flight.

**Key attributes**
- Opportunity reference, Unit reference (the bound unit)
- Template reference (which standard plan this is based on, or "custom")
- Total amount, currency, total milestones count
- Modification history (renegotiations with reasons and timestamps)
- Special terms notes

**Relationships**
- BELONGS_TO: Opportunity (sales pipeline only)
- HAS_MANY: PaymentMilestones

**States**
- Draft, Active, Renegotiating, Completed, Defaulted, Cancelled

---

## 19. PaymentMilestone

A single payment due within a PaymentPlan. Has its own due date, amount, trigger condition, status, and reminder rules.

**Key attributes**
- PaymentPlan reference
- Sequence number, name (deposit, stage 1, etc.), description
- Amount, currency, due date
- Trigger condition (date-based, or event-based like "on completion of structural works")
- Status, paid amount, payment date, payment confirmation Document reference
- Reminder rules, escalation rules
- Linked Notification IDs for reminders sent

**Relationships**
- BELONGS_TO: PaymentPlan
- HAS_MANY: Documents (receipts, proof of payment)

**States**
- Pending, Upcoming (approaching due date), Due, Paid, Overdue, Waived, Disputed

---

## 20. Document

A polymorphic file entity with type-specific extensions. Each document type has its own lifecycle and attributes. Attached to Contact, Opportunity, Project, PaymentMilestone, or other addressable entities.

**Universal attributes (Document parent)**
- File reference (storage URL), filename, format, size
- Document type discriminator
- Subject reference (polymorphic)
- Uploaded by User, upload timestamp
- Version number, supersedes Document reference (for revised documents)
- Retention policy reference
- Access scope (who can see this within the Workspace)

**Type-specific extensions**

- **KYCDocument**: regulator scheme satisfied, verification authority, expiry date, sensitive flag (extra access controls), country of issue
- **ContractDocument**: signatories list, signing platform reference (DocuSign, etc.), signed timestamp, witness references, legal review status
- **ArchitectFile**: drawing type (elevation, plan, section, site plan), revision number, file format (DWG, IFC, PDF), extraction status, ConstraintModel version generated from it
- **PaymentDocument**: payment reference, amount confirmed, currency, payment method, bank reference
- **MarketingBrief / Other**: generic, fewer constraints

**Relationships**
- BELONGS_TO: Workspace
- ATTACHES_TO: any addressable entity
- VERIFIED_BY: User (optional)

**States**
- Per-type state machines. Examples:
  - KYCDocument: Uploaded → Pending_Verification → Verified | Rejected | Expired
  - ContractDocument: Drafted → Out_For_Signature → Signed → Filed | Voided
  - ArchitectFile: Uploaded → Extracting → Extracted | Extraction_Failed → Superseded
  - PaymentDocument: Uploaded → Verified → Filed

## 21. Commission

- BELONGS_TO: Workspace
- PAYABLE_TO: User or Partner
- DERIVED_FROM: Opportunity

- LINKS_TO: Documents (payment receipts)

## 26. Offer

A formal proposal of terms made within an Opportunity. Captures price, payment terms, conditions, and validity. Supports counter-offer chains, multi-unit bulk offers, and internal approval workflows. Polymorphic across pipeline types: richest structure in the sales pipeline (units, payment plans, finishes), lighter structure in other pipelines (e.g., pre-buy land pipeline offers on plots). Most relevant near-term scope is sales.

**Key attributes**
- Opportunity reference (the deal context)
- Submitted by (polymorphic: Contact for buyer offers, User for developer counter-offers)
- Offer type (initial, counter, revised, final)
- References previous Offer (the offer this one counters, if any) for counter-offer chain reconstruction
- Subjects covered (polymorphic by pipeline type):
  - Sales pipeline: one or more Units (supports bulk offers across multiple units)
  - Pre-buy land pipeline: Plot or Site reference (future)
  - Other pipeline types: as defined by the Pipeline
- Price terms: offered amount per subject, total amount, currency
- Payment terms (sales pipeline): proposed PaymentPlan template or custom schedule, deposit amount, deposit terms
- Conditions: included finishes upgrades, parking allocation, storage, custom modifications, completion timing requests, contingencies
- Validity: submitted at, expires at, time-to-respond
- ApprovalRequest reference (if terms fall outside list-price guardrails or require management sign-off)
- Counterparty response, response timestamp, response notes
- Linked Documents (signed offer sheet, term sheet, supporting correspondence)

**Relationships**
- BELONGS_TO: Opportunity
- COVERS: one or more Units (sales pipeline) or other subjects depending on Pipeline type
- REFERENCES: previous Offer (for counter-offer chains)
- REQUIRES: ApprovalRequest (when internal approval is needed)
- LINKS_TO: Documents
- BECOMES_BASIS_FOR: PaymentPlan and Reservation (when Accepted, in sales pipeline)

**States**
- Drafted, Submitted, Under_Internal_Review, Counter_Proposed, Accepted, Rejected, Withdrawn, Expired, Superseded (by a later offer in the chain)

## 27. ConsultationSession

A facilitated, agent-led conversation between Bricly and a User about a decision the User wants help thinking through. The on-demand consultant pattern. Captures the back-and-forth, the surfaced references (mood boards, style examples, market positioning, prior-project patterns, performance data, comparable-deal history), the recommendations made by the agent, and the decisions the user landed on. Produces structured outputs that feed downstream Tools (Briefs, price updates, campaign changes, assignment rule changes, etc.) depending on the consultation type. Distinct from Activity (which logs what happened) and Brief (which is finalised generation input). A ConsultationSession is the human-facing reasoning layer that produces the inputs to downstream actions, and the audit trail for why a decision was made.

Available at any point in the developer's workflow. Used at launch for the project sense-check and brand consultation. Used by marketing leads for campaign direction, channel mix, and budget decisions. Used by managers for pricing decisions, lead allocation rules, and performance reviews. Used by C-level for cross-project pattern review and strategic decisions. Same entity, type discriminator scopes the conversation focus and the context loaded.

**Key attributes**
- Type (project_launch, brand_consultation, microsite_consultation, campaign_consultation, pricing_consultation, positioning_consultation, channel_strategy, budget_allocation, lead_allocation_rules, performance_review, cross_project_pattern_review, ad_hoc, other)
- Subject reference (polymorphic, optional: Project, Opportunity, Campaign, BrandKit, User, or Workspace for cross-project decisions)
- Conversation log (ordered messages between User and agent, with timestamps, role, content, attached references)
- Context inputs loaded at start (varies by type): for project_launch and brand_consultation, the full CRM project intake (name, developer, type, sale status, construction status, completion, price range, location, total units, floors, parking, property types, bedrooms, amenities, key selling points, target buyer, availability with unit detail, brochure if existing, payment terms, mortgage availability, all uploaded files including drawings/CAD/floor plans/availability/plot photos) plus the Studio brief intake (style direction, brand stage, timeline, references) plus ConstraintModel extraction output plus corporate brand plus prior-project patterns. For other types, the relevant data slice (pipeline state, performance metrics, campaign attribution, payment history, etc.)
- Surfaced references during the session (mood boards, sample renders, example BrandKits from prior projects, positioning notes, market data, performance comparables, with links to where each came from)
- Agent recommendations made during the session
- User decisions captured at conclusion (vary by type: style tags, target buyer profile, brand inheritance setting, tone of voice, price changes, budget allocations, assignment rules, anything to explicitly avoid)
- Depth budget (soft cap on questions, defaults vary by type, generally 4-8 for creative decisions, can be more for complex strategic ones; adaptive based on context confidence)
- Resulting outputs (Briefs created, Tool invocations triggered, tags added to entities, preference profile updates, price updates, campaign changes, assignment rule changes)
- Status, started at, concluded at, abandoned at (if applicable), facilitating Agent reference

**Relationships**
- BELONGS_TO: Workspace, optionally Project (or Campaign, or other subject)
- FACILITATED_BY: User (the human in the conversation), Agent (the internal Bricly agent)
- PRODUCES: Briefs, tags on Contact/Project/Unit/Opportunity, preference profile updates, price changes, campaign updates, assignment rule changes (polymorphic by type)
- LINKS_TO: created Assets (mood boards generated during the session), prior ConsultationSessions for cross-reference, related performance reports or data exports
- INFORMS: BrandKit selection, Campaign direction, pricing decisions, positioning, budget allocation, lead allocation rules

**States**
- Active, Paused, Completed, Abandoned

**Status: out of MVP and v1 scope. Sketched here to ensure the data model can extend without refactoring.**

The entities below cover the post-sale journey: reservation, contracts, completion tracking, buyer-facing portal, and multi-party collaboration with notaries and solicitors. # End of Future Expansion Layer

The five entities above represent the post-sale digitisation layer. They are out of MVP scope. The current ideal-product model (entities 1-27) does not depend on them, and they have been designed to plug in cleanly:
- Document polymorphism already supports ContractDocument, payment receipts, completion certificates
- ApprovalRequest, Notification, Appointment, and AuditEvent all naturally extend to cover post-sale workflows
- Microsite can be extended to include the white-label buyer portal as a new type, or PurchaserPortalAccess can replace it in this layer
- PaymentPlan and PaymentMilestone already cover the financial side of completion

**Strategic note**: post-sale digitisation is a real competitive area (Unlatch and similar platforms are established here). Bricly's strategic question is whether to extend into post-sale to become the end-to-end AI-OS for developers, or to integrate with existing post-sale platforms and stay focused on the pre-sale and sale stages. The data model accommodates either path.

## Cross-cutting relationships

A few relationship patterns worth being explicit about, because they cross multiple entities and matter for the capability surface:

**The personalisation chain**: sales-pipeline Opportunity → Brief (personalisation) → Asset (personalised) → published via Microsite (personalised_buyer_pack) → distributed → linked back to Opportunity for tracking.

**The sync chain**: Unit status change → SyncEvent created → re-evaluation of Assets that reference this Unit → stale Assets flagged → regeneration Briefs queued → Microsites refreshed → new Assets replace old → Notifications fired to relevant Users → audit trail preserved in AuditEvent log.

**The attribution chain**: Campaign → first capture point on Contact (capture cluster) → Opportunity created with source reference → Partner (if applicable, with commission split) → on close, Commission records allocated to User and Partner → ApprovalRequests for payout → Payment Documents filed.

**The launch package generation chain**: Project enters Launching state after onboarding → brand consultation Mode A part 1 produces Project.name, Building names, Personas, BrandKit.tone_of_voice → Phase 0 renders generate (4 Assets, neutrally styled, vision-verified) → consultation resumes for Mode A part 2 producing BrandKit.art_direction → Mode B generates 4 BrandKit candidates, developer selects → Phase 2a fires (4 style-approval renders, brand-applied) → Phase 2b on approval (full baseline render set + multi-face hero adders) → Phase 2c (FinishPackage variant interior renders, one set per Active FinishPackage) → Phase 2d (FaceTransitionVideo Assets via Kling orchestration, multi-face buildings) → AssetUnitMaps proposed on exterior renders and level floor plans → ContentBackbone composed from Project, Building, Unit, FinishPackage, BrandKit, Persona references → brochure (print + digital) and Microsite (project_public_site, Draft) generated in parallel from the backbone → marketing material templates (Canva-orchestrated) and launch-grid SocialPost generated → all Approved Assets surface in CRM project, unit, opportunity views via Resource queries (R12, R17, R18) → Project remains in Launching state, awaiting the Go-to-Market flow.

**The constraint chain**: every Brief references the current ConstraintModel version. Every Asset records which ConstraintModel version it was generated under. Vision verification on every Generated render checks layer 1 attributes against the ConstraintModel; three silent regenerations on failure, then Brief state Escalated_To_Internal. When ConstraintModel updates (architect revision through ArchitectFile re-upload), Assets generated under old versions are flagged Stale and a SyncEvent propagates regeneration. Layer 1 is not developer-overridable; only the architect can revise it.

**The consultation chain**: ConsultationSession is the on-demand consultant pattern, callable at any decision point in the developer's workflow. At project launch, the CRM project intake (name, developer, type, sale status, construction status, completion, price range, location, total units, floors, parking, property types, bedrooms, amenities, key selling points, target buyer, availability list, brochure, payment terms, mortgage availability, all uploaded files) plus the Studio brief intake (style direction, brand stage, timeline, references) plus ConstraintModel extraction plus corporate brand all become context. Agent runs a focused, adaptive Q&A within the soft cap, surfaces mood boards and recommendations, captures decisions, and produces the downstream Brief(s) and Tool invocations (`generate_brand_kit_options`, microsite generation, campaign direction). Outside the launch flow, the same pattern serves marketing leads (campaign direction, channel mix, budget allocation), managers (pricing decisions, lead allocation rules, performance reviews), and C-level (cross-project pattern review, strategic decisions). Each session persists as the audit and reasoning record for the decision, feeding D6 (cross-project pattern learning) as decision history compounds.

**The approval chain**: ApprovalRequest can target any approvable entity. Same audit pattern, same queue, same notification logic across holds, commission, collateral, exceptions.

**The interactive mapping chain**: Architect floor plan uploaded as Asset → extraction pipeline parses unit labels → AssetUnitMap rows created joining Asset to Units, polygons populated, low-confidence mappings flagged for review → on the public Microsite or in the personalised buyer pack, polygons render as interactive zones → buyer clicks a polygon → deep link to Unit detail or personalised pack → Activity logged on the Opportunity. When Asset is regenerated, related AssetUnitMaps go Stale and re-mapping is required.

**The multi-pipeline Contact chain**: a single Contact may have concurrent Opportunities across multiple Pipelines (e.g., marketing nurture + active sales pursuit). Pipeline automation rules can fire cross-pipeline SyncEvents (e.g., when an Opportunity in the sales pipeline transitions to Closed_Won, an automation rule pauses or transitions the linked marketing-pipeline Opportunity for the same Contact).

**The payment chain**: Opportunity reaches Reserved or Contract_Signed → PaymentPlan instantiated from a template (or from an Accepted Offer's terms) → PaymentMilestones scheduled → as due dates approach, Notifications fire → buyer pays → PaymentDocument uploaded and verified → PaymentMilestone marked Paid → SyncEvent propagates to reporting → if commission-triggering milestone, Commission record created or updated.

**The offer chain**: Rep aligns with buyer on terms → Offer drafted → submitted for internal approval if outside guardrails → ApprovalRequest fires to manager → manager approves, rejects, or counter-proposes → if counter-proposed, new Offer created referencing the previous one (chain link) → buyer responds → on Accepted, the Offer becomes the source of truth for deal terms → PaymentPlan instantiates from the Offer's payment terms → Reservation references the Offer → Commission calculations use the Offer's agreed amount, not list price. The full negotiation history is preserved as a linked chain of Offer records.

## What's deliberately not here

This is the ideal-product model. Things deliberately excluded:

- **Lead as a separate entity.** Considered, rejected. Contact handles everything. Segmentation by purpose (marketing leads, sales leads, professionals) is a UI concern handled through tabs, smart lists, and filtered views on the Contact entity. State and contact-type tags on Contact carry the semantic meaning.

- **Role as a separate entity.** Folded into User as an attribute cluster (role enum + permission scope + assignment rules). May promote to its own entity if fine-grained permissions get genuinely complex, but not in the ideal model.

- **ExternalProfessional as a separate entity.** Notaries, lawyers, finance brokers, surveyors are handled as Contacts with `contact_type = external_professional` and a sub-type field. Promote to its own entity if their behaviour diverges meaningfully from Contacts.

- **Channel as a separate entity.** Lead source is captured on Contact (capture cluster) and on Opportunity (source field). Paid channels are handled by Campaign. Organic, walk-in, referral, and other non-paid sources are enum values. Channel-as-entity would over-model what is mostly attribution metadata.

- **Agent / ToolInvocation / McpClient as data model entities.** These are platform-layer concerns. Agency MCP clients are Partners. External buyer agents are session-level. Tool invocations log as AuditEvents.

- **NotificationPreferences as a separate entity.** Lives as a cluster on User.

- **CorporateBrand as a separate entity.** Lives as a cluster on Workspace.
